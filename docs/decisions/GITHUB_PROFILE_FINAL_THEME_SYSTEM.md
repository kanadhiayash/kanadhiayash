## Final decision

The profile uses three visual layers with separate responsibilities:

1. **Minimal Samurai Light** for identity, hero, metrics, spacing, typography, and recruiter-facing presentation.
2. **Minimal Alien-tech** for Zeref architecture, benchmarks, evidence maps, system flows, and technical proof.
3. **Hacker Console** as an optional small technical accent only.

Core rule:

> Samurai identity, Alien-tech evidence, Hacker detail.

The three styles must not be used everywhere at once.

## Samurai Light rules

Use for:

- hero;
- clickable metric portals;
- overall page rhythm;
- static tool rail;
- recruiter-facing labels and navigation.

Visual language:

- warm parchment;
- restrained crimson;
- minimal ink wash;
- sparse cherry blossoms;
- disciplined spacing;
- clean modern typography.

Avoid:

- cosplay imagery;
- excessive Japanese text;
- fake calligraphy;
- decorative unreadable seals;
- dense ornament;
- aggressive motion.

Approved Japanese phrase:

**改善・構築・守る**  
**Improve · Build · Protect**

The seal uses a verified **YK monogram**, not invented Japanese lettering.

## Alien-tech rules

Use for:

- Zeref system portal;
- architecture;
- benchmarks;
- memory and continuity flows;
- evidence and guardrail modules.

Visual language:

- light background;
- restrained deep green;
- circuit-line geometry;
- orbital nodes;
- controlled technical framing;
- clear evidence labels.

Avoid fake system data, unreadable HUD density, alien character art, and novelty graphics.

## Hacker rules

Hacker styling is optional and limited to one small technical detail, such as:

- a benchmark trace;
- a terminal status strip;
- a compact system console;
- a hidden technical route.

It must never define Yash's primary identity.

## Interaction rules

GitHub-native interaction only:

- linked SVGs or GIFs;
- anchors;
- linked images;
- expandable `<details>` evidence drawers.

No JavaScript, cursor tracking, audio, drag interaction, or fake multi-zone image controls.

Each animated component must lead to useful evidence.

## Motion budget

First production version:

1. one animated hero;
2. three metric portal animations;
3. one Zeref evidence animation.

Product GIFs are added only after verified product media exists.

Motion must be slow, functional, optional, and understandable without animation.

## Accessibility

Every moving asset requires:

- descriptive alt text;
- reduced-motion handling;
- a static fallback;
- accessible text outside the image;
- no flashing faster than three times per second;
- no color-only meaning.

## Evidence controls

Do not change or exaggerate:

- project ownership;
- team attribution;
- limitations;
- status;
- evidence routes;
- security boundaries;
- non-claims.

Do not introduce fabricated screenshots, users, revenue, deployments, benchmarks, certification status, or seniority claims.

## Approval and merge gate

- Work through a review branch and pull request.
- Run repository quality, link, accessibility, and release gates.
- Review light mode, dark GitHub surroundings, desktop, and narrow width.
- Do not merge without explicit approval from Yash Kanadhia.
